# CLAUDE.md 更新指南

## 核心原则

只添加真正能帮助未来 Claude 会话的信息。上下文窗口是宝贵的——每行都必须证明自己存在的价值。

## 应该添加的内容

### 1. 发现的命令/工作流

```markdown
## 构建

`npm run build:prod` — 带优化的完整生产构建
`npm run build:dev` — 快速开发构建（无压缩）
```

原因：避免未来会话再次发现这些。

### 2. 陷阱和非显而易见的模式

```markdown
## 陷阱

- 由于共享数据库状态，测试必须串行运行（`--runInBand`）
- `yarn.lock` 是权威的；如果依赖不匹配，删除 `node_modules`
```

原因：防止重复调试。

### 3. 包之间的关系

```markdown
## 依赖关系

`auth` 模块依赖 `crypto` 先初始化。
导入顺序在 `src/bootstrap.ts` 中很重要。
```

原因：从代码中无法看出的架构知识。

### 4. 有效的测试方法

```markdown
## 测试

API 端点测试：使用 `supertest` 配合 `tests/setup.ts` 中的测试辅助函数
Mock 策略：使用 `tests/factories/` 中的工厂函数（而非内联 mock）
```

原因：建立有效的模式。

### 5. 配置上的特殊之处

```markdown
## 配置

- `NEXT_PUBLIC_*` 变量必须在构建时设置，而非运行时
- Redis 连接需要添加 `?family=0` 后缀以支持 IPv6
```

原因：环境特定的知识。

## 不应添加的内容

### 1. 代码中显而易见的

不好的例子：
```markdown
`UserService` 类处理用户操作。
```

类名已经告诉我们这些。

### 2. 通用最佳实践

不好的例子：
```markdown
始终为新功能编写测试。
使用有意义的变量名。
```

这是通用建议，非项目特定。

### 3. 一次性修复

不好的例子：
```markdown
我们在提交 abc123 中修复了一个登录按钮不工作的 bug。
```

不太可能再次发生；只会让文件变得臃肿。

### 4. 冗长的解释

不好的例子：
```markdown
认证系统使用 JWT 令牌。JWT（JSON Web Token）是一种开放标准（RFC 7519），
定义了一种紧凑且自包含的方式，用于在各方之间以 JSON 对象的形式安全传输信息。
在我们的实现中，我们使用 HS256 算法...
```

好的例子：
```markdown
认证：JWT + HS256，令牌放在 `Authorization: Bearer <token>` 头部。
```

## 更新的 Diff 格式

对每个建议的变更：

### 1. 标识文件

```
文件：./CLAUDE.md
章节：命令（在 ## 架构 之后的新章节）
```

### 2. 展示变更

```diff
 ## 架构
 ...

+## 命令
+
+| 命令 | 用途 |
+|---------|---------|
+| `npm run dev` | 带 HMR 的开发服务器 |
+| `npm run build` | 生产构建 |
+| `npm test` | 运行测试套件 |
```

### 3. 说明原因

> **为什么有帮助：** 构建命令未记录，导致不清楚如何运行项目。
> 这避免未来会话需要去查看 `package.json`。

## 验证清单

在最终确定更新前，验证：

- [ ] 每条新增都是项目特定的
- [ ] 没有通用建议或显而易见的信息
- [ ] 命令已测试并能工作
- [ ] 文件路径准确
- [ ] 新的 Claude 会话会觉得这有帮助吗？
- [ ] 这是表达该信息最简洁的方式吗？
