# CLAUDE.md 改进器（中文版）

审计、评估和改进代码库中的 CLAUDE.md 文件，确保 Claude Code 拥有最优的项目上下文。

**本 skill 可以写入 CLAUDE.md 文件。** 在输出质量报告并获得用户批准后，通过针对性改进来更新 CLAUDE.md 文件。

## 工作流

### 阶段1：发现

查找仓库中所有 CLAUDE.md 文件：

```bash
find . -name "CLAUDE.md" -o -name ".claude.md" -o -name ".claude.local.md" 2>/dev/null | head -50
```

**文件类型与位置：**

| 类型 | 位置 | 用途 |
|------|------|---------|
| 项目根目录 | `./CLAUDE.md` | 主要项目上下文（纳入 git 版本管理，团队共享） |
| 本地覆盖 | `./.claude.local.md` | 个人/本地设置（已 gitignore，不共享） |
| 全局默认 | `~/.claude/CLAUDE.md` | 跨项目的用户级默认配置 |
| 包专用 | `./packages/*/CLAUDE.md` | monorepo 中的模块级上下文 |
| 子目录 | 任意嵌套位置 | 功能/领域专用上下文 |

**注意：** Claude 会自动发现父目录中的 CLAUDE.md 文件，使 monorepo 配置自动生效。

### 阶段2：质量评估

对每个 CLAUDE.md 文件，对照质量标准进行评估。详见 [references/quality-criteria.md](references/quality-criteria.md) 评分细则。

**快速评估清单：**

| 标准 | 权重 | 检查项 |
|-----------|--------|-------|
| 命令/工作流已记录 | 高 | 是否包含构建/测试/部署命令？ |
| 架构清晰度 | 高 | Claude 能否理解代码库结构？ |
| 非显而易见的模式 | 中 | 是否记录了陷阱和坑？ |
| 简洁性 | 中 | 没有冗长解释或显而易见的信息？ |
| 时效性 | 高 | 是否反映当前代码库状态？ |
| 可操作性 | 高 | 指令是否可执行而非模糊？ |

**质量评分：**
- **A (90-100)**：全面、最新、可操作
- **B (70-89)**：覆盖良好，有少量缺口
- **C (50-69)**：基本信息，缺少关键部分
- **D (30-49)**：稀疏或过时
- **F (0-29)**：缺失或严重过时

### 阶段3：输出质量报告

**在进行任何更新之前，务必先输出质量报告。**

格式：

```
## CLAUDE.md 质量报告

### 概要
- 找到的文件数：X
- 平均分：X/100
- 需要更新的文件数：X

### 逐个文件评估

#### 1. ./CLAUDE.md（项目根目录）
**得分：XX/100（等级：X）**

| 标准 | 得分 | 备注 |
|-----------|-------|-------|
| 命令/工作流 | X/20 | ... |
| 架构清晰度 | X/20 | ... |
| 非显而易见的模式 | X/15 | ... |
| 简洁性 | X/15 | ... |
| 时效性 | X/15 | ... |
| 可操作性 | X/15 | ... |

**问题：**
- [列出具体问题]

**建议新增：**
- [列出应添加的内容]

#### 2. ./packages/api/CLAUDE.md（包专用）
...
```

### 阶段4：针对性更新

输出质量报告后，征得用户确认再进行更新。

**更新指南（关键）：**

1. **仅建议有针对性的新增**——聚焦于真正有用的信息：
   - 在分析过程中发现的命令或工作流
   - 在代码中发现的陷阱或非显而易见模式
   - 之前不明确的包关系
   - 有效的测试方法
   - 配置上的特殊之处

2. **保持最小化**——避免：
   - 复述代码中显而易见的内容
   - 已经涵盖的通用最佳实践
   - 不太可能再次发生的一次性修复
   - 当一行就够时却使用冗长解释

3. **展示 diff**——对每个变更，展示：
   - 要更新哪个 CLAUDE.md 文件
   - 具体新增内容（以 diff 或引用块形式）
   - 简要说明为什么这对未来会话有帮助

**Diff 格式：**

```markdown
### 更新：./CLAUDE.md

**原因：** 缺少构建命令，导致不清楚如何运行项目。

\`\`\`diff
+ ## 快速开始
+
+ ```bash
+ npm install
+ npm run dev  # 在端口 3000 启动开发服务器
+ ```
\`\`\`
```

### 阶段5：应用更新

在用户批准后，使用 Edit 工具应用变更。保留现有内容结构。

## 模板

详见 [references/templates.md](references/templates.md) 中各项目类型的 CLAUDE.md 模板。

## 常见问题标记

1. **过时命令**：不再有效的构建命令
2. **缺少依赖**：未提及的必要工具
3. **过时的架构**：已更改的文件结构
4. **缺少环境设置**：必要的环境变量或配置
5. **损坏的测试命令**：已变更的测试脚本
6. **未记录的陷阱**：未捕获的非显而易见模式

## 用户提示

在呈现建议时，提醒用户：

- **`#` 键快捷方式**：在 Claude 会话期间，按 `#` 键让 Claude 自动将所学内容纳入 CLAUDE.md
- **保持简洁**：CLAUDE.md 应该可读；紧凑胜于冗长
- **可操作的命令**：所有记录的命令应可直接复制粘贴
- **使用 `.claude.local.md`**：个人偏好不与团队共享（添加到 `.gitignore`）
- **全局默认值**：将跨用户偏好放在 `~/.claude/CLAUDE.md`

## 优秀的 CLAUDE.md 的特征

**关键原则：**
- 简洁且可读
- 可复制粘贴的可操作命令
- 项目特定模式，而非通用建议
- 非显而易见的陷阱和警告

**推荐章节**（仅使用相关的部分）：
- 命令（构建、测试、开发、lint）
- 架构（目录结构）
- 关键文件（入口点、配置）
- 代码风格（项目约定）
- 环境（必需变量、设置）
- 测试（命令、模式）
- 陷阱（坑、常见错误）
- 工作流（何时做什么）
