# feature-workflow 共识文档

> 本文档记录 feature-workflow 设计与实现规范的所有讨论共识。
> 所有决策均通过 grill-me 流程达成，每个决策包含问题、选项、最终选择和理由。

---

## 决策记录

### 决策 1：阶段模型

**状态**：已达成共识

**问题**：feature-workflow 的阶段应该如何划分？

**选项**：
- A. 维持原五阶段（理解→计划→审批→执行→验证）
- B. 扩展为七阶段（增加接口设计和接口审批）
- C. 用户提出的新模型

**最终选择**：用户提出并确认的新模型

```
理解需求(grill-me) → 接口设计(grill-me) → 接口审批 → 分步实现 → 验证收尾
```

**理由**：
- 接口审批后契约冻结，实现阶段主要是填充内部逻辑
- grill-me 作为深度对齐方法论，需要在需求和接口两个关键节点分别执行
- 传统"计划审批"被 grill-me 流程替代，因为"理解了需求，接口文件自然就会产生"

### 决策 2：理解需求与接口设计的边界

**状态**：已达成共识

**问题**：两个连续的 grill-me 阶段（理解需求、接口设计）的边界在哪里？

**选项**：
- A. 理解需求只产出文字摘要，接口设计再产出代码
- B. 理解需求直接产出接口代码骨架，接口设计阶段对齐分支和约束
- C. 两个阶段合并为一个

**最终选择**：B

**理由**：
- "理解了需求，接口文件自然就会产生，用接口代码去说明需求"
- 需求理解和接口草案是同一个阶段的产出，不应拆成两次文档
- 接口设计阶段专门对齐所有异常分支、边界条件和设计约束

**关键约束**：
- 接口的参数、异常、返回值都可以占位，不需要太过深入讨论
- 但所有分支必须对齐（异常、边界、空值、权限等）

---

### 决策 3：数据结构在接口阶段的位置和形态

**状态**：已达成共识

**问题**：接口代码骨架中，数据结构（DTO/Entity/VO）放在什么位置？在接口阶段应该定义到什么程度？

**选项**：
- A. 接口阶段只写函数签名，不写数据结构
- B. 接口阶段写数据结构骨架（核心字段确定，其余标 //TODO）
- C. 接口阶段完整定义数据结构

**最终选择**：B

**理由**：
- "好的数据结构可以让人一看就懂业务逻辑"——接口阶段需要数据结构来支撑理解
- "数据结构可以在实现阶段动态增减字段"——接口阶段不应定死所有字段
- 数据结构名称是接口契约的一部分，代表一组状态的集合；具体字段是封装细节

**关键约束**：
- 数据结构属于接口契约的一部分，和接口文件放在同一层级
- 核心字段在接口阶段确定，用于支撑主要功能点和至少一个测试用例

### 决策 4：使用说明文件与接口代码的分工

**状态**：已达成共识

**问题**：既然接口代码就是需求的具现化，为什么还需要额外的使用说明文件？两者的分工是什么？

**选项**：
- A. 使用说明是"用户视角"设计文档，接口代码是"开发者视角"契约
- B. 使用说明只是接口代码的聚合索引
- C. 不需要额外使用说明文件，接口代码 + 测试用例 = 完整说明

**最终选择**：A

**理由**：
- 使用说明回答"为什么这样设计"和"典型调用链路"，代码只能展示"是什么"
- 使用说明本身也是设计文档，内部实现可以不写
- 测试用例展示"怎么用"，但无法解释"为什么这么设计"

**关键约束**：
- 单模块的使用说明，最终形式必须是一份独立文件
- 除非是纯库 API，否则代码注释（如 Javadoc）不能替代独立文件

---

### 决策 5：测试用例在接口阶段的形态

**状态**：已达成共识

**问题**：测试用例在接口阶段应该是什么形态？

**选项**：
- A. 只创建测试骨架（空方法 + 场景注释）
- B. 创建可编译但会失败的测试（TDD 风格）
- C. 不创建独立测试文件，测试用例就是使用说明中的伪代码

**最终选择**：B 的变体：至少写一个核心测试（可编译但失败），其余可为骨架

**理由**：
- "至少写一个测试用例，确定主要功能点，确定方向，防止走歪"
- 基于已确定的核心数据结构字段，可以写出有意义的断言
- 执行起来必然失败，符合 TDD 的 Red 状态
- 实现阶段补充完整后，测试通过

**关键约束**：
- 接口阶段至少一个基于核心字段的测试用例
- 其余测试方法可为空壳 + 场景注释
- Web 界面模块使用《界面测试步骤说明》作为测试用例交付物

### 决策 6：接口审批的机制

**状态**：已达成共识

**问题**：接口审批应该审什么？谁来审？

**选项**：
- A. 用户审，只审业务正确性
- B. AI 自检 + 用户兜底确认，只审技术约束
- C. 分成两轮——AI 技术自检后，用户业务终审

**最终选择**：C

**理由**：
- 用户可能不懂技术约束（如 doSomething 原则、impl 隔离），无法判断是否符合工程规范
- AI 可能误解业务意图，但自检全绿，用户简单兜底无法发现
- 两轮审批兼顾技术规范和业务正确性

**AI 自检清单**：
1. 模块单一性：只有一个自包含主功能点
2. doSomething 原则：API 以业务动词为主
3. 层级隔离：有 impl/internal 包规划
4. 分支对齐：异常、边界、空值、权限已考虑
5. 使用说明：存在独立文件
6. 测试用例：至少一个核心测试
7. 数据结构契约：名称体现业务语义，核心字段已定义

### 决策 7：分步实现的执行模式

**状态**：已达成共识

**问题**：接口审批通过后，实现阶段应该按什么模式执行？

**选项**：
- A. 按子功能点逐个验收（细粒度，用户每一步确认）
- B. 一次性实现整个模块（粗粒度）
- C. 按批次实现（中粒度）

**最终选择**：用户提出的新模式——半自动模式

**新模式定义**：
- AI 根据接口契约、使用说明、测试用例，重新生成实现计划
- AI 按子功能点逐个静默实现并自动 commit
- 用户无需每一步确认，主打无人值守
- 但允许用户随时介入细粒度审查

**理由**：
- "接口定义确定后，AI 已经受到很大的约束了"
- "对齐需求和接口以后，其他都是内部实现，不重要"
- "换一个上下文，根据接口、文档、测试用例，重新生成计划，然后静默执行"

**自动暂停条件**：
1. 测试失败且 AI 无法自动修复
2. AI 发现必须突破契约约束
3. 实现中发现接口设计假设不成立

### 决策 8：实现阶段的契约冻结约束

**状态**：已达成共识

**问题**：接口审批通过后，实现阶段哪些可以改，哪些绝对不能改？

**最终选择**：以下约束清单

**绝对禁止**：
- 不许修改对外暴露的 API 方法签名和名称
- 不许修改 DTO/数据结构名称
- 不许删除核心字段
- 不许将内部辅助类扶正为对外 API

**允许**：
- 新增非核心字段
- 新增内部辅助类/方法，但必须放在 internal/impl 包中

**理由**：
- 接口审批后契约冻结，对外契约不可变
- 内部实现有调整空间，但必须封装隔离
- 如果确实需要新的对外 API，除非后期有专门的重构任务，否则不允许

**内部辅助类规则**：
- 实现中需要的新依赖（如 `InventoryService.lock()`）必须放到 `internal` 包中
- 它永远是调用方 API 的一个内部步骤
- 唯一扶正途径：后期重构任务，或需求确定阶段明确提出该模块必须暴露

### 决策 9：前端架构的区分与处理

**状态**：已达成共识

**问题**：模块可以包含界面（条目 1），但条目 4 的接口/实现隔离主要针对后端。前端怎么处理？

**选项**：
- A. 前端没有接口/实现之分，整个前端代码都是"实现"
- B. 前端组件拆分——Props 为接口，内部 State 为实现
- C. 模块的"接口"仅指后端 API，前端页面只是消费者

**最终选择**：按架构类型区分处理

**全栈一体式框架**（MVC、Vaadin 等）：
- 前端代码是模块接口的一部分
- 在设计阶段，至少要把代表界面的组件/View/Widget 类创建出来
- 甚至需要渲染出具体界面（不带逻辑的静态界面）
- 后端 API 的包装接口也是广义的接口

**前后端分离架构**：
- 前端是独立应用，不在本模块范围内
- 本模块只定义后端 API
- 因为"不是包装，而是完全就是另一个应用了"

**理由**：
- 全栈一体式中，前端和后端是同一应用的不同层，前端组件直接暴露给用户
- 前后端分离中，前端和后端是两个独立应用，应各自有独立的模块边界

**关键约束**：
- 全栈场景下，接口阶段需要产出前端组件骨架（空壳组件 + 路由 + API 调用骨架）
- 前端内部逻辑不受"接口/实现隔离"约束（因为"前端本地就是接口"）

### 决策 10：验证收尾的测试标准

**状态**：已达成共识

**问题**："实现完成后，具体测试用例必须通过"，这个"通过"的标准是什么？

**选项**：
- A. 所有测试必须自动化通过（包括前端界面）
- B. 后端测试自动化通过，前端界面允许人工验证
- C. 核心路径自动化，边缘场景人工确认

**最终选择**：B

**理由**：
- 后端契约回归风险必须由自动化测试覆盖
- 全栈前端界面的自动化测试依赖 agent + 爬虫 MCP，可靠性不如后端测试框架
- 强求前端自动化可能将 MCP 可用性变成流程阻塞点

**关键约束**：
- 后端单元测试/集成测试必须自动化执行并通过
- 前端界面人工验证，但**必须存在**一份前端文本描述（agent 使用指南）
- 爬虫 MCP 作为可选增强工具，不在 plugin.json 中声明为必备依赖

### 决策 11：项目架构的判定方式

**状态**：已达成共识

**问题**：AI 如何知道当前项目是全栈一体式还是前后端分离？

**选项**：
- A. 完全自动探测
- B. 用户在理解需求阶段声明
- C. 默认前后端分离，高置信度特征时提示切换

**最终选择**：C

**理由**：
- 当前主流是前后端分离，默认按此处理符合大多数场景
- 全栈框架有明显的技术特征（依赖、注解、目录结构），AI 可以高置信度检测
- 完全自动可能误判；每次都问增加用户负担

**判定规则**：
- 默认按前后端分离处理
- 发现 `vaadin-core`、`thymeleaf-spring`、`@Controller` + `*.html` 同包 → 提示全栈一体式
- 用户可在理解需求阶段明确声明，覆盖自动判定

### 决策 12：特性指南文档的定位

**状态**：已达成共识

**问题**：先生成的"特性指南文档"面向谁？共识文档和特性指南的分工是什么？

**选项**：
- A. 只给 AI 看的执行指南
- B. 只给开发者看的设计规范
- C. 两者兼顾

**最终选择**：C（两者兼顾）

**理由**：
- 条目中既有"给 AI 的约束"（如不许改 API 名称），也有"给开发者的设计哲学"（如 doSomething 原则）
- 特性指南前半部分讲原理（给人看），后半部分讲执行步骤和检查清单（给 AI 看）

**文档分工**：
- **特性指南文档**：基于共识整理成的可执行规范，去除了讨论过程，保留最终规则
- **共识文档**：本次 grill-me 讨论中所有决策点的汇总记录（本文档）
