# 特性开发指南

> 本文档定义了 feature-workflow 中模块（特性）的设计与实现规范。
>
> - **前半部分（设计原理）**面向开发者，阐述模块设计的核心思想。
> - **后半部分（执行规范）**面向 AI，提供可执行的步骤、检查清单和模板。

---

## 第一部分：设计原理

### 1. 什么是模块/特性

- **特性即主功能点/模块**，必须只有一条自包含的描述
- **不允许出现两个并列的功能点描述**——那是两个模块，应拆分为独立特性
- **主功能可以包含一组子功能**，子功能组合成特性，这是正常的
- 示例：仓储管理、订单管理、登录模块

### 2. 接口的广义定义

模块对外暴露的功能即接口，不限于狭义的库函数：

| 接口类型 | 示例 |
|---------|------|
| HTTP/Web API | `POST /api/orders` |
| 程序库函数 | `OrderService.createOrder(dto)` |
| 界面组件（全栈一体式） | Vaadin 的 `OrderManagementView`、MVC 的 `order.html` |

模块可以包含界面，也可以不含界面只有 API，甚至可以包含 Web 接口作为 API。

### 3. 三层架构：模块 → 接口 → 实现

```
模块（业务边界）
  └── 接口（对外契约）
      └── 实现（内部逻辑，必须隔离）
```

- **模块**：定义业务功能的边界和范围，回答"这个特性做什么"
- **接口**：定义对外暴露的契约，包括函数签名、数据结构名称、界面结构，回答"外部怎么用它"
- **实现**：填充具体业务逻辑，回答"内部怎么做"，**必须和接口物理隔离**

### 4. API 设计哲学：业务逻辑入口优先

- 程序由**业务逻辑**和**数据结构**组成
- 好的数据结构可以让人一看就懂业务逻辑
- 但 API 一定是以暴露**业务逻辑入口**为主，不是暴露数据结构
- 重点是 `doSomething`，不是直接丢一个对象
- 数据结构体现业务逻辑，是因为业务需要才修改数据结构
- **业务驱动，API 驱动**

**反例 vs 正例：**

```java
// 反例：暴露数据结构字段
user.setName("xxx");
user.setAge(20);
userRepository.save(user);

// 正例：暴露业务操作入口
userService.updateProfile(new UpdateProfileRequest("xxx", 20));
```

### 5. 数据结构作为接口契约

- 数据结构**名称**是接口契约的一部分，代表一组状态的集合
- 状态中具体有什么**字段**，是封装细节，实现阶段可动态增减
- **核心字段**在接口阶段确定，用于支撑主要功能点和至少一个测试用例
- 数据结构作为接口的输入/输出载体，和接口文件放在同一层级（子包或同包）

### 6. 使用说明即设计文档

- 因为模块 → 接口 → 实现三层分明，**使用说明本身也是设计文档**
- 使用说明中具体内部实现可以不写
- 单模块的使用说明，**最终形式必须是一份独立文件，另起文件编写**
- **明确排除 Javadoc、文档字符串等代码内注释**——它们无法替代独立文件
- 使用说明回答：
  - 这个模块解决什么问题
  - 典型调用链路是什么
  - 各接口之间如何协作
  - 有哪些业务约束和已知限制

### 7. 测试用例的双重属性

- 模块**必须**包含测试用例
- 测试用例本身也是使用说明，给人看了就知道模块怎么用
- 文本说明也是一份测试用例（只是不自动化执行而已）
- Web 界面的测试可以用 agent 结合自动化爬虫去测试
- **实现完成后，具体测试用例必须通过**

### 8. 什么是 grill-me

grill-me 是一种**深度对齐方法论**，用于在关键设计节点确保所有决策分支都被充分讨论、所有依赖关系都被厘清、所有潜在漏洞都被暴露。

#### grill-me 的执行规则

1. **逐层深入**：沿着设计决策树的每一个分支逐层深入，不跳过任何层级
2. **逐个厘清**：逐个厘清不同决策之间的依赖关系
3. **持续追问**：针对计划的方方面面进行持续追问，直到达成共识
4. **每次一问**：每次只提出一个问题，避免信息过载
5. **推荐答案**：每个问题都附上推荐答案，供参考和讨论
6. **代码探查**：如果某个问题可以通过探查代码库得到答案，直接查阅代码库，无需提问
7. **用户决策**：最终决策权在用户，AI 可提议但不可强制

#### grill-me 在本流程中的适用场景

- **理解需求阶段**：以用户为拷问对象，将模糊的业务需求转化为明确的模块边界和接口代码骨架
- **接口设计阶段**：以接口代码为拷问对象，确保接口契约的完备性，对齐所有异常分支和边界条件

## 第二部分：执行规范（面向 AI）

### 阶段模型

```
理解需求(grill-me) → 接口设计(grill-me) → 接口审批 → 分步实现 → 验证收尾
```

---

### 阶段一：理解需求（grill-me）

#### 目标
将模糊的业务需求转化为明确的模块边界和**可编译的接口代码骨架**。

#### 执行步骤

1. **读取需求源**
   - 通过平台 MCP 读取 Issue 正文和评论
   - 读取项目 README 和相关模块入口文件，确认模块边界
2. **启动 grill-me**
   - 以用户为拷问对象，按 grill-me 执行规则逐层对齐
   - 核心拷问点：
     - 这个特性是否只有一个自包含的主功能点？
     - 主功能包含哪些子功能？
     - 模块是否需要界面？是全栈一体式还是纯后端 API？
     - 有哪些业务约束和技术约束？
3. **输出接口代码骨架**
   - 函数签名（参数/返回值可用占位符或核心类型）
   - 数据结构骨架（核心字段确定，其余标 `//TODO`）
   - 异常声明（异常类型可占位，但必须有）
   - 前端组件骨架（全栈一体式）：空壳组件 + 路由配置 + API 调用骨架

#### 输出物清单

- [ ] 接口代码文件（后端 API）
- [ ] 数据结构文件（核心字段已定义）
- [ ] 前端组件骨架（全栈场景）
- [ ] 《需求理解摘要》（ grill-me 共识记录）

---

### 阶段二：接口设计（grill-me）

#### 目标
确保接口契约的完备性，**对齐所有分支**。

#### 执行步骤

1. **基于接口代码骨架启动 grill-me**
   - 以接口代码为拷问对象，按 grill-me 执行规则逐层对齐
   - 核心拷问点：
     - 所有异常分支是否已考虑？（空值、越界、权限不足、并发冲突）
     - 边界条件是否已定义？
     - 是否符合 doSomething 原则？（API 是否是业务动词）
     - 数据结构名称是否体现业务语义？
2. **为每个接口和数据结构添加语言级注释**
   - 每个接口方法必须带注释：说明业务语义、参数含义、返回值含义、异常场景
   - 每个数据结构（DTO/VO/Entity）必须带注释：说明该结构代表的业务状态集合
   - 注释使用各自语言的标准文档语法（Java 用 `/** */`、TypeScript 用 `/** */`、Python 用 `"""` 等）
   - 这些注释是代码级别的说明，**不能替代独立的使用说明文件**
3. **补充伪代码和文字描述**
   - 在接口方法体内用 `//TODO` 和伪代码描述具体实现逻辑
   - 确保接口完成时，模块至少在 QA 逻辑层面跑得通
4. **创建测试骨架**
   - 至少写一个基于核心字段的测试用例（可编译但会失败）
   - 其余测试方法可为空壳 + 场景注释

#### 输出物清单

- [ ] 对齐后的接口代码（含伪代码和 //TODO）
- [ ] 《使用说明》文件（用户视角的设计文档）
- [ ] 测试用例文件（至少一个核心测试）
- [ ] 前端文本描述（全栈场景：给 agent 使用的界面指南）

---

### 阶段三：接口审批

接口审批分为两轮：

#### 第一轮：AI 技术自检

| 检查项 | 通过标准 |
|--------|---------|
| 模块单一性 | 只有一个自包含主功能点，无并列功能点 |
| doSomething 原则 | API 以业务动词为主，不暴露数据结构字段 |
| 层级隔离 | 实现类规划在 `impl`/`internal` 包或带 `Impl` 后缀 |
| 分支对齐 | 异常、边界、空值、权限等已考虑 |
| 使用说明 | 存在独立的《使用说明》文件 |
| 测试用例 | 至少一个可编译但失败的核心测试 |
| 数据结构契约 | 名称体现业务语义，核心字段已定义 |

**注意**：前端组件在全栈一体式框架下属于接口契约的一部分，也需要被审查。

#### 第二轮：用户业务终审

- 用户检查：接口是否覆盖了需求中的所有子功能点
- 用户检查：调用链路是否符合业务直觉
- 用户检查：前端界面骨架（全栈场景）是否符合预期
- **用户确认后方可进入实现阶段**

---

### 阶段四：分步实现

#### 执行模式：半自动

- AI 根据接口契约、使用说明、测试用例，**重新生成实现计划**
- AI 按子功能点逐个静默实现并自动 commit
- **用户无需每一步确认**，实现阶段主打无人值守

#### 自动暂停条件

遇到以下情况时 AI **必须自动暂停**，等待用户指示：

1. 测试失败且 AI 无法自动修复
2. AI 发现必须突破契约约束（如确实需要新增对外 API 方法）
3. 实现中发现接口设计假设不成立（如伪代码中的依赖不存在）

#### 契约冻结约束（绝对禁止突破）

- **不许修改**对外暴露的 API 方法签名和名称
- **不许修改** DTO/数据结构名称
- **不许删除**核心字段
- **只允许**新增非核心字段
- **允许新增**内部辅助类/方法，但必须放在 `internal`/`impl` 包中，**不允许暴露**

#### 内部实现的新增规则

如果实现中需要新的辅助依赖（如 `InventoryService.lock()`）：
- 该辅助类必须放到 `internal` 包中
- 它永远是调用方 API 的一个内部步骤
- 除非后期有专门的重构任务，否则不允许将其扶正为对外 API

---

### 阶段五：验证收尾

#### 后端测试（所有架构通用）

- 单元测试/集成测试必须自动化执行并通过
- 测试覆盖率以核心路径为准，不强制要求 100% 行覆盖

#### 前端验证（仅全栈一体式）

- 后端测试通过后，AI 展示前端界面渲染结果
- 用户人工验证交互逻辑和视觉效果
- **必须存在**一份前端文本描述（agent 使用指南），即使当前使用人工验证

#### Issue/PR 收尾

- 在 Issue 中添加完成评论，包含变更汇总
- 如有遗留项，标注为后续任务
- 向用户展示 Draft PR URL，提示最终审核

## 第三部分：目录结构规范

### 后端模块（Java 示例）

```
com.example.module.order/
├── api/
│   ├── OrderService.java          # 对外接口
│   ├── OrderDTO.java              # 数据结构契约
│   └── package-info.java          # 模块说明（可选）
├── internal/                      # 实现隔离
│   ├── OrderServiceImpl.java      # 接口实现
│   ├── OrderRepository.java       # 内部数据访问
│   └── InventoryService.java      # 内部辅助类（不暴露）
├── docs/
│   └── README.md                  # 模块使用说明
└── ...（测试文件在 src/test 对应包下）
```

### 全栈一体式模块（Vaadin/MVC 示例）

```
com.example.module.order/
├── api/
│   ├── OrderService.java          # 后端接口
│   └── OrderDTO.java              # 数据结构
├── view/                          # 前端界面组件（接口的一部分）
│   └── OrderManagementView.java   # 或 order.html（MVC）
├── internal/                      # 后端实现隔离
│   ├── OrderServiceImpl.java
│   └── OrderRepository.java
├── docs/
│   ├── README.md                  # 模块使用说明
│   └── FRONTEND_GUIDE.md          # 前端文本描述（agent 指南）
└── ...
```

### 前后端分离模块

- 本模块只包含后端 API 接口和实现
- 前端是独立应用，不在本模块范围内
- 目录结构同"后端模块"

### 模板 1：需求理解摘要

```markdown
## 需求理解摘要

- **主功能点**: {一条自包含的描述}
- **子功能列表**:
  1. {子功能 1}
  2. {子功能 2}

- **已知约束**:
  - 技术: {性能/兼容性/版本约束}
```

## 附录：项目架构判定

AI 在进入理解需求阶段前，自动判定项目架构：

1. **扫描项目特征**：
   - 发现 `vaadin-core`、`thymeleaf-spring`、`@Controller` + `*.html` 同包 → 提示全栈一体式
   - 发现独立的 `frontend/`、`src/App.tsx`、根目录 `package.json` → 前后端分离
   - 无明显前端代码 → 纯后端

2. **默认策略**：
   - 默认按**前后端分离**处理
   - 检测到高置信度全栈特征时，主动提示用户确认

3. **用户覆盖**：
   - 用户可在理解需求阶段明确声明架构类型，覆盖自动判定结果
