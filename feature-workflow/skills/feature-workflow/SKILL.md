# feature-workflow: Feature 开发规范化流程

## 核心原则

1. **阶段不可跳过**：每个流程必须按顺序执行，前一个阶段未完成时，提示用户确认成果并询问是否进入下一阶段。
2. **生命周期自包含**：每一个阶段都可以根据前一个阶段产出的文档独立执行，不依赖任何会话上下文或记忆。
3. **文档驱动**：每个阶段必须有明确的文档产出，文档是阶段间的唯一契约和 RAG 上下文来源。
4. **契约冻结**：接口审批通过后，对外暴露的 API 契约不可变，实现阶段只允许填充内部逻辑。

---

## 什么是 grill-me

grill-me 是一种**深度对齐方法论**，用于在关键设计节点确保所有决策分支都被充分讨论、所有依赖关系都被厘清、所有潜在漏洞都被暴露。

### grill-me 的执行规则

1. **逐层深入**：沿着设计决策树的每一个分支逐层深入，不跳过任何层级
2. **逐个厘清**：逐个厘清不同决策之间的依赖关系
3. **持续追问**：针对计划的方方面面进行持续追问，直到达成共识
4. **每次一问**：每次只提出一个问题，避免信息过载
5. **推荐答案**：每个问题都附上推荐答案，供参考和讨论
6. **代码探查**：如果某个问题可以通过探查代码库得到答案，直接查阅代码库，无需提问
7. **用户决策**：最终决策权在用户，AI 可提议但不可强制

### grill-me 在本流程中的适用场景

- **理解需求阶段**：以用户为拷问对象，将模糊的业务需求转化为明确的模块边界和接口代码骨架
- **接口设计阶段**：以接口代码为拷问对象，确保接口契约的完备性，对齐所有异常分支和边界条件

---

## 阶段模型

```
理解需求(grill-me) → 接口设计(grill-me) → 接口审批 → 分步实现 → 验证收尾
```

每个阶段 transition 时，AI 必须：
1. 展示当前阶段产出物摘要
2. 明确提示："{阶段}已完成，成果如上。是否进入{下一阶段}？"
3. **等待用户确认**，不允许自动推进

---

## 阶段一：理解需求（grill-me）

### 目标
将模糊的业务需求转化为明确的模块边界和**可编译的接口代码骨架**。

### 执行步骤

1. **读取需求源**
   - 通过平台 MCP 读取 Issue 正文和评论
   - 读取项目 README 和相关模块入口文件，确认模块边界
   - 判定项目架构类型（全栈一体式 / 前后端分离），默认前后端分离，检测到高置信度全栈特征时提示用户确认
2. **启动 grill-me**
   - 以用户为拷问对象，按 grill-me 执行规则逐层对齐
   - 核心拷问点：
     - 这个特性是否只有一个自包含的主功能点？
     - 主功能包含哪些子功能？
     - 模块是否需要界面？是全栈一体式还是纯后端 API？
     - 有哪些业务约束和技术约束？
3. **生成需求文档**
   - 将所有模糊需求明确化后的共识写入 `docs/feature/{模块名}-需求.md`
   - 文档包含：模块边界、主功能点、子功能列表、业务约束、技术约束、待澄清问题（已解决）

### 输出物

- `docs/feature/{模块名}-需求.md`（需求理解摘要 + grill-me 共识）
- **阶段一不生成任何接口代码骨架或函数签名**

---

## 阶段二：接口设计（grill-me）

### 目标
确保接口契约的完备性，**对齐所有分支**。

### 执行步骤

1. **读取需求文档**
   - 读取 `docs/feature/{模块名}-需求.md`，获取模块边界、子功能列表和约束
2. **确定目录结构**
   - 基于项目依赖和语言生态，确定模块的目录结构
   - 原则：接口与实现隔离（如 `api/` + `internal/` 或语言惯例等价结构）
   - 数据结构作为接口契约，和接口文件同层级
   - **不在 SKILL.md 中显式声明固定目录结构**，由阶段二根据项目实际情况确定
3. **启动 grill-me**
   - 以接口草案为拷问对象，按 grill-me 执行规则逐层对齐
   - 核心拷问点：
     - 所有异常分支是否已考虑？（空值、越界、权限不足、并发冲突）
     - 边界条件是否已定义？
     - 是否符合 doSomething 原则？（API 是否是业务动词）
     - 数据结构名称是否体现业务语义？
4. **生成接口代码骨架**
   - 基于已确认的需求，创建接口文件、数据结构文件
   - 函数签名（参数/返回值可用占位符或核心类型）
   - 数据结构骨架（核心字段确定，其余标 `//TODO`）
   - 异常声明（异常类型可占位，但必须有）
   - 前端组件骨架（全栈一体式）：空壳组件 + 路由配置 + API 调用骨架
5. **为每个接口和数据结构添加语言级注释**
   - 每个接口方法必须带注释：说明业务语义、参数含义、返回值含义、异常场景
   - 每个数据结构（DTO/VO/Entity）必须带注释：说明该结构代表的业务状态集合
   - 注释使用各自语言的标准文档语法（Java 用 `/** */`、TypeScript 用 `/** */`、Python 用 `"""` 等）
   - **这些代码级注释不能替代独立的设计文档**
6. **补充伪代码和文字描述**
   - 在接口方法体内用 `//TODO` 和伪代码描述具体实现逻辑
   - 确保接口完成时，模块至少在 QA 逻辑层面跑得通
7. **创建测试骨架**
   - 至少写一个基于核心字段的测试用例（可编译但会失败）
   - 其余测试方法可为空壳 + 场景注释
8. **生成设计文档**
   - 将接口设计成果写入 `docs/feature/{模块名}-设计.md`

### 输出物

- `docs/feature/{模块名}-设计.md`（接口设计文档，同时承担使用说明职责）
- 接口代码骨架（后端 API + 数据结构 + 前端组件）
- 测试用例文件（至少一个核心测试）
- 前端文本描述（全栈场景：给 agent 使用的界面指南）

### 设计文档规范

- **明确排除 Javadoc、文档字符串等代码内注释**——它们无法替代独立文件
- **必须另起文件编写**，最终形式是一份独立的 Markdown 文件
- 内容同时承担**设计文档**和**使用说明**的双重职责：
  - 这个模块解决什么问题
  - 典型调用链路是什么
  - 各接口之间如何协作
  - 有哪些业务约束和已知限制
  - 前端界面结构（全栈场景）

## 阶段三：接口审批

接口审批分为两轮：

### 第一轮：AI 技术自检

| 检查项 | 通过标准 |
|--------|---------|
| 模块单一性 | 只有一个自包含主功能点，无并列功能点 |
| doSomething 原则 | API 以业务动词为主，不暴露数据结构字段 |
| 层级隔离 | 实现类规划在 `impl`/`internal` 包或带 `Impl` 后缀 |
| 分支对齐 | 异常、边界、空值、权限等已考虑 |
| 设计文档 | 存在独立的 `docs/feature/{模块名}-设计.md` |
| 测试用例 | 至少一个可编译但失败的核心测试 |
| 数据结构契约 | 名称体现业务语义，核心字段已定义 |
| 语言级注释 | 每个接口和数据结构都有标准注释 |

**注意**：前端组件在全栈一体式框架下属于接口契约的一部分，也需要被审查。

AI 自检完成后，向用户展示自检结果（通过/未通过项）。如有未通过项，返回阶段二修复。

### 第二轮：用户业务终审

- 用户检查：接口是否覆盖了需求中的所有子功能点
- 用户检查：调用链路是否符合业务直觉
- 用户检查：前端界面骨架（全栈场景）是否符合预期
- **用户确认后方可进入下一阶段**

### 审批通过后生成计划文档

用户确认后，AI 立即生成 `docs/feature/{模块名}-计划.md`，内容包含：
- 子功能点列表（从接口代码提取）
- 执行顺序（考虑依赖关系）
- 批次划分（哪些可并行、哪些必须串行）
- 预计涉及的文件清单

**计划文档需要用户确认后方可执行。**

## 阶段四：分步实现

### 前置条件
- `docs/feature/{模块名}-计划.md` 存在且用户已确认
- 接口契约已冻结

### 执行模式：半自动

- AI 读取计划文档和接口源码，按子功能点逐个静默实现并自动 commit
- 用户无需每一步确认，主打无人值守
- 但允许用户随时通过"暂停"命令介入细粒度审查

### 实时维护执行记录

每完成一个子功能点，AI 在 `docs/feature/{模块名}-执行.md` 中实时追加/更新：

```markdown
## 执行记录：{模块名}

**计划开始**: {timestamp}
**当前进度**: {已完成数}/{总数}

| 序号 | 子功能点 | 状态 | Commit | 问题/备注 |
|------|---------|------|--------|----------|
| 1 | {名称} | 完成 | `{hash}` | 无 |
| 2 | {名称} | 进行中 | — | 等待测试修复 |
```

### 自动暂停条件

遇到以下情况时 AI **必须自动暂停**，等待用户指示：

1. 测试失败且 AI 无法自动修复
2. AI 发现必须突破契约约束（如确实需要新增对外 API 方法）
3. 实现中发现接口设计假设不成立（如伪代码中的依赖不存在）

### 契约冻结约束（绝对禁止突破）

- **不许修改**对外暴露的 API 方法签名和名称
- **不许修改** DTO/数据结构名称
- **不许删除**核心字段
- **只允许**新增非核心字段
- **允许新增**内部辅助类/方法，但必须放在 `internal`/`impl` 包中，**不允许暴露**

### 内部实现的新增规则

如果实现中需要新的辅助依赖（如 `InventoryService.lock()`）：
- 该辅助类必须放到 `internal` 包中
- 它永远是调用方 API 的一个内部步骤
- 除非后期有专门的重构任务，否则不允许将其扶正为对外 API

## 阶段五：验证收尾

### 后端测试（所有架构通用）

- 单元测试/集成测试必须自动化执行并通过
- 测试覆盖率以核心路径为准，不强制要求 100% 行覆盖

### 前端验证（仅全栈一体式）

- 后端测试通过后，AI 展示前端界面渲染结果
- 用户人工验证交互逻辑和视觉效果
- **必须存在**一份前端文本描述（agent 使用指南），即使当前使用人工验证

### 生成验收文档

AI 生成 `docs/feature/{模块名}-验收.md` 草稿：

```markdown
## 验收报告：{模块名}

- **议题**: #{N}
- **验收时间**: {timestamp}

### 实现范围
- [x] {子功能 1}
- [x] {子功能 2}
- [ ] {子功能 3}（原因: {延期/取消}）

### 测试结果
- 后端测试：{通过/失败}
- 前端验证：{通过/人工确认}

### 变更汇总
- 新增文件：{列表}
- 修改文件：{列表}
- 提交记录：{列表}

### 遗留项
- {如有}

### 用户确认
- [ ] 确认完成
- [ ] 存在遗留项，需后续处理
```

**用户确认后定稿**，AI 在 Issue 中添加完成评论，并提示 Draft PR URL。

## 状态恢复与上下文切换

### 核心机制

技能可以中途退出，也可以换一个上下文继续。恢复状态的唯一方式是**探测文件系统**——不依赖会话记忆。

### 探测顺序

AI 接收到用户指令（如"继续"或"开始任务"）时，按以下顺序探测 `docs/feature/` 目录：

| 探测条件 | 推断状态 | AI 动作 |
|---------|---------|---------|
| `{模块名}-验收.md` 存在 | 已完成 | 展示验收结果，询问是否需要 reopen |
| `{模块名}-执行.md` 存在且未标记完成 | 执行阶段 | 读取执行记录，汇报当前进度，询问是否继续 |
| `{模块名}-计划.md` 存在且执行.md 不存在 | 计划已生成 | 展示计划摘要，询问用户是否确认并开始执行 |
| `{模块名}-设计.md` 存在且计划.md 不存在 | 接口设计完成 | 提示用户进行人工审批，审批通过后生成计划 |
| `{模块名}-需求.md` 存在且设计.md 不存在 | 需求阶段完成 | 展示需求文档，询问是否进入接口设计 |
| 都不存在 | 从头开始 | 进入理解需求阶段 |

### 模块定位方式

1. **自动探测**：AI 扫描 `docs/feature/` 目录和当前 git 分支，列出候选模块
2. **用户确认**：用户选择或确认目标模块
3. **优雅降级**：如果自动探测失败，要求用户显式指定模块名或议题号

### 阶段过渡行为

当 AI 探测到某阶段已完成，但用户未明确说"进入下一阶段"时，AI 必须：
- 展示该阶段产出物摘要（读取对应文档的核心内容）
- 明确提示："{阶段}已完成，成果如上。是否进入{下一阶段}？"
- **等待用户确认，不允许自动推进**

## 文档产出规范

| 阶段 | 文档 | 位置 | 命名 | 时机 |
|------|------|------|------|------|
| 需求理解 | 需求文档 | `docs/feature/` | `{模块名}-需求.md` | 阶段一完成时 |
| 接口设计 | 设计文档 | `docs/feature/` | `{模块名}-设计.md` | 阶段二完成时 |
| 接口审批 | 计划文档 | `docs/feature/` | `{模块名}-计划.md` | 审批通过后，执行前 |
| 分步实现 | 执行记录 | `docs/feature/` | `{模块名}-执行.md` | 实时追加 |
| 验证收尾 | 验收报告 | `docs/feature/` | `{模块名}-验收.md` | 用户确认后定稿 |

**所有文档必须是 Markdown 格式。**

## 操作限制

### 绝对禁止（无例外）

- 未获得用户阶段过渡确认，自动推进到下一阶段
- 在接口审批通过前修改对外 API 契约
- 在实现阶段删除核心字段或修改 DTO 名称
- 将 `internal`/`impl` 包中的辅助类扶正为对外 API
- 在本 workflow 中执行 `git push --force` / `git push -f`

### 需用户确认（AI 可提议，用户批准后执行）

- 修改已通过审批的接口契约（新增/删除 API 方法、调整签名）
- 跳过某个子功能点的实现
- 对非 feature 类型议题强制进入本 workflow
- 并行执行多个子功能点（需要用户确认批次划分）

### 有条件可执行

- **合并多子功能点为一个 commit**：仅当同时满足以下条件时 AI 可提议，用户确认后执行：
  1. 多个子功能点属于同一原子模块，拆分会导致 commit 不完整
  2. 用户在计划确认时已知晓并同意合并
  3. 合并后的 commit 消息清晰列出所有实现的功能点

## 附录：项目架构判定

AI 在进入理解需求阶段前，自动判定项目架构：

1. **扫描项目依赖**（优先于文件夹扫描）：
   - 读取 `pom.xml` / `build.gradle` / `build.gradle.kts`：发现 `vaadin-core`、`thymeleaf-spring`、`spring-boot-starter-web` + 无独立前端构建配置 → 提示全栈一体式
   - 读取根目录 `package.json`：发现独立前端框架依赖（`react`、`vue`、`angular`）且存在独立的 `vite.config.js` / `webpack.config.js` → 前后端分离
   - 同时存在后端构建文件和前端构建文件，但前端构建配置在独立目录（如 `frontend/package.json`）→ 前后端分离
   - 无明显前端依赖 → 纯后端

2. **默认策略**：
   - 默认按**前后端分离**处理
   - 从依赖文件中检测到高置信度全栈框架依赖时，主动提示用户确认

3. **用户覆盖**：
   - 用户可在理解需求阶段明确声明架构类型，覆盖自动判定结果
