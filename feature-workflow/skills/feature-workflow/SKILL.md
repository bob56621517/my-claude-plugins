# feature-workflow: Feature 开发规范化流程

## 触发条件

`git-context-prep` 完成 feature 类型议题的环境初始化后，自动进入本工作流。

## 流程概览

```
理解需求 → 制定计划 → 用户审批 → 分步执行 → 验证收尾
```

---

## 阶段一：理解需求

1. **读取需求源**
   - 读取 Issue 正文、评论、关联 PR
   - 确认模块边界和用户期望
2. **输出**：需求理解摘要，包括模块定位、功能列表、已知约束

## 阶段二：制定计划

1. **模块拆解**
   - 将需求拆解为平级 API 列表
   - 每项 API 标注：名称、描述、输入/输出、影响范围
2. **依赖检查**
   - 前置 API 是否存在
   - 外部依赖（MCP 工具、第三方库）是否就绪
3. **计划模板**

```
模块：{模块名}
├── API-1：{名称} — {描述}
├── API-2：{名称} — {描述}
└── API-N：{名称} — {描述}

前置依赖：
- {依赖项 1}：[就绪/缺失]

影响范围：{涉及的文件/目录列表}
预估工作量：{估计}
```

## 阶段三：用户审批

1. 展示完整计划
2. 等待用户确认或提出修改意见
3. 用户确认后进入执行阶段

## 阶段四：分步执行

1. **按 API 维度串行执行**
   - 每步实现一个 API
   - 每步完成后展示当前状态，等待用户验收
2. **执行规范**
   - 每次 commit 遵循 `#议题号 实现 {API 名}` 格式
   - 非阻塞性问题记录为 Issue 评论，不打断当前流程
3. **用户可在任意步骤提出修改**
   - 当前步骤暂停 → 修改 → 重新提交验收

## 阶段五：验证收尾

1. **完整性验证**
   - 对照计划逐一确认所有 API 已实现
2. **Issue 状态更新**
   - 关闭 Issue 或添加完成评论
3. **变更汇总**
   - 列出所有新增/修改的文件
   - 列出所有 commit
