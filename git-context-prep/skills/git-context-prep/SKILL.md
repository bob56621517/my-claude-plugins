# git-context-prep: 议题/任务上下文准备

git-context-prep skill 只在"开始或切换议题/任务"时触发。纯 git 操作（commit/push/merge/rebase/pull）不依赖本 skill。

## 两个入口

```
入口 A: "开始任务 X"     → 搜索议题
                           ├── 搜索到 → 确认 → 走入口 B
                           └── 没搜到 → 创建新议题 → 完整新建流程
入口 B: "继续 #N"        → 基于已有议题跳转
```

---

## Step 1：远端操作（仅 MCP / 平台 API）

远端平台操作，语义目标描述，AI 自行选择平台 MCP 或 CLI 实现。不执行任何 git CLI 命令。若无 MCP 可用，引导用户手动在平台网页查看并输入结果。

### 入口 A：开始任务

#### A1. 搜索议题

AI 从用户描述提取关键词，搜索开放议题。

```
├── 搜索到 → 列出 top 3-5 个（标题 + 编号 + 标签），用户选择确认
│            → 走入口 B 流程（基于已有议题继续）
│
└── 没搜到 → AI 生成标题+描述+推断标签，展示确认 → 创建新议题
             → 继续 A2
```

创建议题字段：标题、描述、标签（AI 推断）

#### A2. 分支类型确认

AI 从 feature/fix/docs/refactor/test/chore 推断（不限于所列类型），用户确认。

#### A3. 选择分支基线

平台 API 列出远端分支：默认分支 + 最近活跃的 15 条分支（排除临时分支，如匹配 `*_temp_*`、`*_tmp_*`、`*_wip_*` 的分支），用户选择从哪条基线分出。

#### A4. 基于基线创建远端分支

平台 API 基于所选基线在远端创建分支：`{议题号}-{类型}-{描述}`

#### A5. 创建 Draft PR（远端）

按平台分叉：

- **GitHub**: `draft: true`，描述含 `Closes #N`。提示用户 PR 合并后如需自动删分支，前往 repo Settings > Merge button 开启 auto-delete head branches
- **GitLab**: 标题前缀 `Draft:`，描述含 `Closes #N`，检查项目级删分支配置，未开启则提示
- **Gitee**: 不支持 Draft → 创建普通 PR 并提示，描述含 `Closes #N`，删分支不支持则告知

#### A6. 记录信息

记录传给 Step 2：议题号（纯数字）、远端分支名、基线分支、Draft PR URL

### 入口 B：继续 #N

#### B1. 检查议题状态

已关闭则提示是否 reopen。

#### B2. 搜索远端分支

平台 API 搜索匹配议题号的远端分支。优先级：精确匹配 `{N}-*` 分支名 → 从已知 PR 的 source branch 提取 → 模糊搜索 → 失败则允许用户手动输入。搜不到 → 回退：搜索标签、关键词，或允许用户手动输入分支名。

#### B3. 搜索 PR

平台 API 检查关联开放 PR/MR。

#### B4. 路由判断

```
├── 分支有 + PR 有
│   展示基线（PR 的 base branch）→ 确认 → 传给 Step 2
│
├── 分支有 + PR 无
│   选择基线（同A3）→ 确认 → 创建 Draft PR（同A5）→ 传给 Step 2
│
└── 分支无
    选择基线（同A3）→ 远端创建分支（同A4）→ 创建 Draft PR（同A5）→ 传给 Step 2
```

## Step 2：本地同步（纯 git CLI）

无 MCP 介入。Step 1 完成后，把远端分支同步到本地。

### 2.1 基线对齐检查（前置步骤）

Step 1 只操作远端 API，不能保证本地基线是最新的。无论当前分支是不是目标开发分支，本地同步前必须先做基线对齐检查。

1. `git fetch origin` — 拉取所有远端引用，不修改本地分支
2. 判断本地基线是否落后于远端基线：
   ```bash
   git rev-list --count --left-right origin/{基线分支}...{基线分支}
   ```
   输出格式如 `<behind>\t<ahead>`（如 `3\t0` 表示本地落后 3 个提交）。
3. 方向判断结论处理（仅记录，不做实际同步）：
   - **behind > 0 且 ahead = 0**：记录「基线过时」结论，供后续冲突方向判断使用。
   - **ahead > 0**：提示用户"本地基线有新提交，是否先 push 到远端？"
   - **behind = 0 且 ahead = 0**：基线同步，无需额外操作。

> 基线分支（合并目标）默认为 main，在 A3/B4 由用户选择确认。开发分支（议题对应）是从基线分出的工作分支，与基线是不同分支，不做相互对比。

### 2.2 流程图（有序列表形式）

1. **`git fetch origin`**（已完成，见 2.1 前置步骤）
2. **有 dirty state？**
   - 是 → 执行脏区处理（见 2.3），处理完才继续
   - 否 → 继续步骤 3
3. **检查本地开发分支是否存在**
   - 本地不存在 → `git switch --create {分支} origin/{分支}` → 上下文准备完成
   - 本地已存在 → 继续步骤 4
4. **方向判断 + 拉取最新代码**
   - 执行方向判断（含日期对比和分叉检查，见 2.4.0）：
     - **远端领先且本地无新提交** → `git pull --ff-only origin {开发分支}` → 完成
     - **本地领先远端** → 跳过 pull，提示用户"本地有远程没有的提交" → 完成
     - **双方分叉** → `git pull origin {开发分支}` → 无冲突完成，有冲突执行冲突处理（见 2.4）

### 2.3 脏区处理（四方案，用户选择）

| 方案 | 命令 | 适用场景 |
|------|------|----------|
| a. 提交当前 | `git add` + `git commit` | 工作已完成，可独立提交 |
| b. 暂存临时分支 | `git switch -c {分支}_temp_{YYMMDD}_{序号}` 后 commit | 内容多/重要，持久保存。如含未跟踪文件，先 `git add -N` 标记或提醒用户处理 |
| c. 本地 stash | `git stash push -m "{标记}"` | 工作未完成，临时切走，回来 pop |
| d. 跳过（不推荐） | 不执行任何操作 | 用户明确要求直接切换，AI 警告后执行。跳过意味着脏区变更会带入后续操作，可能导致合并混乱。AI 展示当前修改文件列表供用户确认 |

未处理脏区前禁止切换分支（选择 d 时除外）。

### 2.4 冲突处理（含方向判断）

`git pull` 产生冲突后，AI **必须先做方向判断，再给建议**。禁止跳过方向判断直接展示 diff 和建议——这会重蹈"错判领先"的覆辙。

#### 2.4.0 方向判断（冲突后第一步动作，不可跳过）

冲突一旦发生，AI 应立即执行方向判断（只读命令），不要在断点处做任何推测：

```bash
# 查看 ahead/behind 计数（主判断依据）
git rev-list --count --left-right origin/{分支}...HEAD

# 查看哪些具体提交在 HEAD 但不在远端
git log --oneline HEAD ^origin/{分支}

# 查看哪些具体提交在远端但不在 HEAD
git log --oneline origin/{分支} ^HEAD
```

#### 2.4.1 三象限判断

根据 `rev-list` 输出的 `<behind>\t<ahead>` 判断方向：

| 场景 | 判断依据 | 动作 |
|------|----------|------|
| **远程领先** | behind > 0 且 ahead = 0 | 倾向保留远程内容。AI 对每个冲突文件标注"远程（更新）vs 本地（过时）"，默认采用远程版本，同时展示本地修改内容供用户确认是否有必须保留的部分。 |
| **本地领先** | ahead > 0 且 behind = 0 | 暂停合并流程，提示用户："本地有 N 个远程没有的提交，建议先 push 或确认后再处理"。用户确认后继续。 |
| **双方分叉** | behind > 0 且 ahead > 0 | 真正的分叉。AI 逐文件分析冲突块，标注每块的来源方和日期，给出逐块合并建议。不可默认偏袒任何一方。 |

#### 2.4.2 冲突解决辅助步骤

方向判断完成后，AI 按以下步骤辅助：

1. 根据四象限确定的方向，展示冲突文件列表和每文件的方向标注
2. 对每个冲突块，标注来源（本地 HEAD / 远端 origin）和对应日期
3. 提供合并建议（基于方向倾向），供用户参考
4. 用户参考建议手动编辑解决
5. 用户报告已解决后 → `git status` 验证冲突标记已清除 → 提醒用户完成 `git add` + `git commit` 完成 merge

AI 不做 --continue / --abort / --skip，由用户决策后手动执行。待用户手动完成 merge 或 abort 后，提示用户重新运行本 SKILL 继续流程。

## 命名约定

| 对象 | 格式 | 示例 |
|------|------|------|
| 常规分支 | `{议题号（纯数字）}-{类型}-{描述}` | `15-feature-git-workflow-fix` |
| 临时分支 | `{当前分支}_temp_{YYYYMMDD}_{序号}` | `15-feature-git-workflow-fix_temp_20260712_1` |
| commit | `#N 正文` | `#15 优化分支命名` |
| Draft PR 标题 | `WIP: #N {类型}: {简短描述}` | `WIP: #15 feature: git workflow fix` |

推测议题号时，必须向用户确认。

## 操作限制

以下规则定义了 AI 在 Step 2 中的操作边界。分级为：绝对禁止 / 需用户确认 / 有条件可执行。

### 绝对禁止（无例外）

- `git push --force` / `git push -f` — 改写远端历史，不可逆
- `git clean -fd` — 删除未跟踪文件，数据丢失风险

### 需用户确认（AI 可提议，用户批准后执行）

- `git rebase` / `git commit --amend` — 改写本地历史，需用户知情
- `git reset --hard` — 仅在同时满足以下全部条件时 AI 可提议，用户确认后执行：
  1. 方向判断已确认本地落后于远端（远端日期 > 本地日期）
  2. 无 uncommitted 修改（`git status --porcelain` 为空）
  3. 用户明确确认"舍弃本地差异，对齐远端"
  4. --hard 只对准 origin/{分支}，不对准本地引用

### 冲突处理约束

- 冲突发生后，AI 必须先执行方向判断（2.4.0），再给建议
- AI 展示 diff 和合并建议，用户手动解决后通知 AI 验证
- AI 不做 --continue / --abort / --skip，由用户决策。待用户手动完成后，提示重新运行本 SKILL
